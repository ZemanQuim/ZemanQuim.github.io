HTTP通信

HTTP请求

特点

是TCP协议下的一种超文本传输协议，包括几个特点：

  • 无连接，一次连接只处理一种请求，比如客户端发出一个请求，建立连接之后处理，服务端返回响应，客户端断开连接之后，这个请求就结束了~

  • 无状态，请求连接断开之后，不会再有记忆，下一次不会记得上一次的请求内容

  • 媒体独立：只要是客户端或服务端可以处理的数据都可以传输

报文

  • HTTP请求包括请求行（方法+URL+协议）+请求头+空行+请求体

  • 请求头包括一些属性定义 Content-Type、Server、Expire、Content-Length、setcookie、Useragent、Origin、Referer、Date、CacheControl、Accept、Access-Control-Request-Method、Access-Control-Request-Headers、Connection、Cookie、If Not Match、Connection

  • HTTP响应包括响应状态行（协议+状态码+状态码文本描述）+报头+空行+响应报文

  • 报头包括一些属性定义 Access-Control-Allow-Origin、Access-Control-Allow-Credentials、Access-Control-Allow-Methods、Server、Content-Type、Content-Length、Useragent、Connection、Date、Set-Cookie、CacheControl、Expire、ETag

状态码

  • 1XX，继续，请求已接受，继续处理
  • 100 继续
  • 101 切换协议，例如切换到新的HTTP版本协议
  • 2XX，成功
  • 200 响应成功
  • 201 成功请求并创建了新的资源
  • 202 已经接受请求，但没有处理完成
  • 203 非授权信息，请求成功，但是返回的信息不在原始服务器，而是一个副本
  • 204 服务器成功处理，但是没有返回内容
  • 206 服务器成功处理了部分GET请求
  • 3XX，需要进一步操作才能完成请求
  • 304 请求没有被修改
  • 302 临时移动，重定向
  • 305 使用代理，请求的资源需要通过代理来访问
  • 4XX，客户端发生错误
  • 400 客户端语法错误
  • 401 请求用户的身份认证
  • 403 服务端理解客户端的请求，但是拒绝执行
  • 404 服务器无法根据客户端的请求找到资源
  • 5XX，服务端发生错误
  • 500 服务器内部错误
  • 502 充当网管或代理的服务器，从远端服务器收到了一个无效请求
  • 501 服务器不支持请求的功能
  • 505 服务器不支持请求的hTTP协议版本，无法完成请求

HTTP与HTTPS

  • HTTP明文传输协议，不适合传输一些敏感或者保密的数据
  • HTTPS加密传输协议，用于传输保密和敏感的数据，在HTTP下加入了SSL层，在传输过程中客户端与服务端需要验证CA证书，在握手阶段还会协商出一个加密的密钥。

SSL握手

  • Client端请求Https连接，同时发送client_random和支持的加密方式
  • server端返回CA证书和server_random（经过server端私钥加密）
  • 验证CA证书，使用证书中公钥加密的permaster_secret(随机数)
  • client发送permaster_secret至server端，server端通过私钥解密
  • client和server端同时通过client_random / server_random/ premaster_secret生成master secret

CA证书验证

  • server端发来证书+CA的signature之后
  • client获取CA的公钥来解码server证书中的signature
  • 解码完成之后，就确认了signature的正确性，得到message digest和代签名的内容（证书内容）
  • 对签名内容进行哈希，得到message digest
  • 比较两次message digest是否相同来验证证书内容是否有效

TCP协议

TCP和UDP

TCP和UDP都属于传输层的协议

TCP

  • 传输控制协议
  • 传输过程是一对一的，一个客户端对应一个服务端建立连接；
  • 面向连接，双方握手之后建立连接，才能发生请求与响应，请求结束之后必须要通过挥手断开连接；
  • 实现可靠的传输，通过TCP传输的数据可靠、无差错、不重复、并且按序到达；
  • 首部较长，20字节
  • TCP提供全双工通信。TCP允许通信双方的应用进程在任何时候都能发送数据。TCP的两端都有发送缓存和接收缓存。
  • TCP并不关心应用程序一次把多长的报文发送到TCP的缓存中，而是根据对方给出的窗口值和当前网络的拥塞情况来决定一条报文段应包含多少字节
  • HTTP请求传输使用TCP协议

UDP

  • 用户报文协议
  • UDP支持一对一，一对多，多对一，多对多的交互通信
  • 不建立连接也可以发送传输数据，传输完成之后也不需要断开连接；
  • 尽可能交付，但不能保证可靠传输
  • 首部较短，8个字节
  • UDP是面向报文的。发送方对应用程序交下来的报文，在添加首部之后就向下交付IP层。UDP对应用层交下来的报文既不合并也不拆分，而是保留这些报文的边界。也就是说应用层交付多长的报文，UDP就照样发送，即一次发送一个报文。
  • UDP发送的报文程度应该是应用进程给出的。UDP没有拥塞控制 这样网络出现拥塞不会使源主机的发送速率降低。允许网络在拥塞时丢失一些数据的应用是可以的。
  • DNS使用UDP协议

如何实现拥塞控制

某一时间对网络带宽或缓存请求的需求超过了可用范围，就会发生拥塞，网络性能变差，因此需要放置过多的数据注入到网络通道，导致网络过载，这个过程叫做拥塞控制。

TCP如何保证可靠传输

  • 应用数据被分割成TCP认为最适合发送的数据块。
  • 超时重传：当TCP发出一个段后，它启动一个定时器，等待目的端确认收到这个报文段。如果不能及时收到一个确认，将重发这个报文段。
  • TCP给发送的每一个包进行编号，接收方对数据包进行排序，把有序数据传送给应用层。
  • 校验和：TCP将保持它首部和数据的检验和。这是一个端到端的检验和，目的是检测数据在传输过程中的任何变化。如果收到段的检验和有差错，TCP将丢弃这个报文段和不确认收到此报文段。
  • TCP的接收端会丢弃重复的数据。
  • 流量控制：TCP连接的每一方都有固定大小的缓冲空间，TCP的接收端只允许发送端发送接收端缓冲区能接纳的我数据。当接收方来不及处理发送方的数据，能提示发送方降低发送的速率，防止包丢失。TCP使用的流量控制协议是可变大小的滑动窗口协议。
  • 拥塞控制：当网络拥塞时，减少数据的发送。

TCP拥塞控制

TCP通过定义拥塞窗口值的大小来控制拥塞，所谓拥塞窗口值，就是接收端期待接收到的报文数，发送方没收到一个确认，就将拥塞窗口值+1，如果拥塞窗口值大于慢开始门限，就直接将拥塞窗口值设为1，并且将慢开始门限减半

TCP标志字符：

  • ACK确认序号，为1时确认有效；
  • FIN，发端完成发送任务；
  • SYN，新建议一个请求

三次握手：

  • 客户端发送一个SYN=1包给服务端seq=client序号，表示新建连接请求
  • 服务端接受报文之后，传输一个SYN=1的报文给客户端，ack=client+1，seq=server，标识确认连接
  • 客户端接收到报文之后，传输一个SYN=0，seq=client+1，ack=server+1给服务端，表示连接建立，握手结束

为什么三次握手

建立三次握手主要是因为A发送了再一次的确认，那么A为什么会再确认一次呢，主要是为了防止已失效的连接请求报文段又突然传送给B，从而产生了错误。 所谓“已失效的连接请求报文”是这样产生的，正常情况下，A发出连接请求，但是因为连接报文请求丢失而未收到确认，于是A再重传一次连接请求，后来收到了请求，并收到了确认，建立了连接，数据传输完毕后，就释放链接，A共发送了两次连接请求报文段，其中第一个丢失，第二个到达了B，没有“已失效的连接请求报文段”，但是还有异常情况下，A发送的请求报文连接段并没有丢失，而是在某个网络节点滞留较长时间，以致延误到请求释放后的某个时间到达B，本来是一个早已失效的报文段，但是B收到了此失效连接请求报文段后，就误以为A又重新发送的连接请求报文段，并发送确认报文段给A，同意建立连接，如果没有三次握手，那么B发送确认后，连接就建立了，而此时A没有发送建立连接的请求报文段，于是不理会B的确认，也不会给B发送数据，而B却一直等待A发送数据，因此B的许多资源就浪费了，采用三次握手的方式就可以防止这种事情发生，例如刚刚，A不理会B，就不会给B发送确认，B收不到A的确认，就知道A不要求建立连接，就不会白白浪费资源，

四次挥手

  • 客户端发送一个FIN=1的断开连接的TCP报文给服务端，表示客户端不会再发送请求给服务端了
  • 服务端收到FIN包之后会返回一个ACK包给服务端，表示确认连接
  • 服务端发送FIN包给客户端，表示连接断开请求，表示服务端不会再发送请求给客户端了
  • 客户端收到之后，发送ACK包给服务端，表示连接断开，服务器收到ACK包之后断开连接，挥手结束

输入URL后发生了什么

DNS解析

DNS解析的意思就是根据域名解析到对应IP地址，解析的过程是先查询本地的HOST文件，看有没有域名和地址的映射关系，如果没有就去进行迭代查找，首先查找根DNS服务器，如果没有再次查找.comDNS服务器，如果没有就去查找baidu.comDNS服务器，如果哈没有就是查找www.baidu.comDNS服务器查找

DNS劫持

就是在域名解析的过程中，没能够得到正确的域名和IP的对应关系，比如解析错误的域名，解析到错误的页面地址；解析错误的域名，解析到非正常的页面；解析正确的域名，映射到错误的页面地址

HTTP劫持

在域名解析的过程中，解析正确的域名和IP地址，但在服务器返回正确的结果之前，返回给你一定的请求，比如弹出广告什么的。HTTP劫持的预防就是使用SSL协议，在请求传输的过程中使用CA证书进行验证，以保证请求过程的安全和保密。

HTTP请求发送

发送HTTP请求，请求报文包括请求行、请求头、空行、请求体

HTTP请求响应

返回HTTP响应，响应报文包括状态行、报文头、空行、响应体

解析渲染页面

  • 解析HTML，构建DOM树
  • 构建DOM树的过程中，如果碰到JS文件，它会等到JS文件加载完后继续解析DOM，会造成JS阻塞，而JS如果还操作了CSSOM则会等CSSOM执行之后再构建DOM，造成严重的JS阻塞。
  • 解析CSS文件，构建CSSOM树 CSS文件中，每一个样式都可以定义为一个StyleSheet节点对象，共同构成一个CSSOM树来定义DOM节点的样式
  • 二者构建渲染树
  • 浏览器渲染页面
  • 重排与重绘 当DOM的大小和位置没有发生变化，只是DOM节点的样式发生改变，发生的是重绘；当DOM的大小和位置发生了变化，就会发生重排

异步

数据传输及页面渲染大致流程：浏览器自上而下解析html文件，遇到外部引入的css文件会顺次解析加载，遇到引入的img会异步加载，遇到引入js文件顺次加载执行，一般在body节点之前会有请求接口数据的js文件，数据返回后解析到body浏览器就开始渲染页面，让用户看到首屏，之后顺次解析到引入的js文件、css文件依次进行操作，其中可能涉及到为页面中的组件添加交互、替换页面内的数据、重绘组件样式以及对页面布局进行回流。

安全问题

CSRF攻击

概念理解

Cross Site Request Forgery，跨站点请求伪造，攻击者伪装成用户的身份，发送一些恶意请求，被访问的网站误以为是用户自身的请求，认为请求合法，便受到恶意攻击。用户打开浏览器访问网站A，输入用户名和密码进行登录，登录后在同一浏览器中访问页面B，B发出请求要求访问A，访问的时候携带cookie信息，向A发出请求，导致网站A接收到B发出的恶意代码

如何监测CSRF攻击

抓取正常的数据包，去掉Referer字段后重新提交，如果提交仍旧有效，就表明存在CSRF漏洞，类似于伪造一次请求，如果请求成功则说明存在漏洞

如何防御CSRF攻击？

  • 验证Referer字段：记录了HTTP请求的来源地址，验证请求来源可以预防CSRF攻击
  • 添加token并验证：将用户的验证信息不保存在cookie中，而是保存在sessionStorage中，在HTTP请求中添加随机的token，对于请求验证token判断是否接受请求
  • 添加自定义属性进行验证

XSS攻击

通过对网页注入可执行代码且成功地被浏览器执行，达到攻击的目的，形成了一次有效XSS攻击，一旦攻击成功，它可以获取用户的联系人列表，然后向联系人发送虚假诈骗信息，可以删除用户的日志等等

如何预防

  • 服务端设置JavaScript无法读取cookie
  • 服务端存储之前过滤掉JavaScript，iframe，script，JavaScript这些敏感词汇
  • 客户端进行输入检查，防止输入JavaScript、cookie这样的敏感词汇
  • 客户端进行编码转义，比如将URL、\等字符进行特殊转义

sql注入攻击

通过把SQL命令插入到Web表单提交或输入域名或页面请求的查询字符串，最终达到欺骗服务器执行恶意的SQL命令。具体来说，它是利用现有应用程序，将（恶意的）SQL命令注入到后台数据库引擎执行的能力，它可以通过在Web表单中输入（恶意）SQL语句得到一个存在安全漏洞的网站上的数据库，而不是按照设计者意图去执行SQL语句。 比如先前的很多影视网站泄露VIP会员密码大多就是通过WEB表单递交查询字符暴出的，这类表单特别容易受到SQL注入式攻击．

如何防治

  • 进行变量的数据类型和格式的检查，比如用户名采用固定的格式来命名，检查是否包含固定的格式，不包含则拒绝执行sql查询
  • 过滤掉单引号，反斜杠，null，双引号等特殊字符
  • 数据库加密

Dos攻击

http是无状态响应，不保存具体的响应状态，也就是说通信方身份是不验证的，可能遇到假的客户端或服务器，例如服务器不知道客户端，黑客通过发送大量无意义的请求，造成服务器假死拒绝响应，这就是所谓的Dos攻击。

如何防治

  • 使用https协议，在请求过程中验证请求双方的身份信息
